## 二、P0 — 必须立即修复

### P0-1: Agent ID 隔离默认值绕过 (违反系统不变量 #1)

**位置**: `umms-api/src/handlers/ingest.rs:35`, `encoder.rs:100`, `epa.rs:30`

**问题**: 多个 API 端点在未提供 `agent_id` 时静默默认为 `"coder"`:

```rust
// ingest.rs:35
AgentId::from_str(body.agent_id.as_deref().unwrap_or("coder"))
```

**验证**: 成功执行了无 agent_id 的 ingest 请求, 数据被写入 "coder" agent 空间:
```bash
curl -X POST /api/ingest -d '{"text":"test memory without agent_id"}'
# 返回 200 OK, chunks_stored: 1 — 写入了 coder agent
```

**影响**: 任何客户端可在不指定身份的情况下读写 "coder" agent 的私有数据, 完全绕过隔离机制。

**修复方案**:
1. 将 `agent_id` 设为必填参数 (`required` 而非 `Option`)
2. 缺少 `agent_id` 时返回 `400 Bad Request`
3. 受影响端点: `/api/ingest`, `/api/encode`, `/api/epa/analyze`

**预计工时**: 2 小时

### P0-2: 巩固引擎 API 返回 500 Internal Server Error

**位置**: `umms-api/src/handlers/consolidation.rs`

**问题**: `POST /api/consolidation/run/coder` 返回 500 错误:
```json
{"error":"internal server error"}
```

**分析**: `ConsolidationScheduler::run_cycle()` 调用失败, 但错误信息被泛化为 "internal server error", 无法诊断根因。可能原因:
- CozoDB graph store 初始化问题
- 向量数据不足导致 WKD 聚类失败
- 配置参数不匹配

**修复方案**:
1. 在 `run_consolidation` handler 中返回结构化错误 (不要用 `String` 作为 error type)
2. 添加 tracing span 记录失败阶段 (decay/wkd/evolution/promotion)
3. 为空数据场景添加 graceful 降级

**预计工时**: 4 小时

### P1-2: LanceDB 查询字符串拼接风险

**位置**: `umms-storage/src/vector/lance_store.rs:133-394`

**问题**: 所有 LanceDB 过滤条件使用 `format!()` 字符串拼接:
```rust
format!("agent_id = '{}'", agent_id.as_str())
format!("id = '{}'", id.as_str())
```

**风险**: 如果 AgentId 或 MemoryId 包含单引号, 可能导致查询注入。当前 Newtype 没有对输入进行字符验证。

**修复方案**:
1. 在 `AgentId::from_str` 和 `MemoryId::from_str` 中添加字符白名单验证 (仅允许 `[a-zA-Z0-9_-]`)
2. 或者在 format 之前 escape 单引号
3. 长期: 如果 LanceDB 支持参数化查询, 迁移到参数化模式

**预计工时**: 2 小时

### P1-3: 审计日志类型安全缺失

## 五、架构级优化建议

### A-1: 引入统一 API 错误类型 (优先级: 高)

**现状**: Handler 返回 `Result<Json<T>, String>`, axum 将 `String` 错误统一转为 500。

**目标架构**:
```rust
pub enum ApiError {
    BadRequest { message: String },
    NotFound { resource: String, id: String },
    Internal { message: String, trace_id: String },
    ServiceUnavailable { service: String },
}

impl IntoResponse for ApiError {
    fn into_response(self) -> Response {
        let (status, body) = match self {
            ApiError::BadRequest { message } => (StatusCode::BAD_REQUEST, json!({"error": message})),
            ApiError::NotFound { resource, id } => (StatusCode::NOT_FOUND, json!({"error": format!("{resource} {id} not found")})),
            ApiError::Internal { message, trace_id } => (StatusCode::INTERNAL_SERVER_ERROR, json!({"error": message, "trace_id": trace_id})),
            ApiError::ServiceUnavailable { service } => (StatusCode::SERVICE_UNAVAILABLE, json!({"error": format!("{service} unavailable")})),
        };
        (status, Json(body)).into_response()
    }
}
```

**收益**: 所有端点返回一致的错误格式, 前端可统一处理, 运维可通过 trace_id 追踪问题。

---

### A-2: Agent ID 验证中间件 (优先级: 高)

**现状**: 每个 handler 独立解析 agent_id, 部分有默认值, 部分报错格式不同。

**目标架构**:
```rust
// axum extractor
pub struct ValidAgentId(pub AgentId);

#[async_trait]
impl<S> FromRequestParts<S> for ValidAgentId {
    type Rejection = ApiError;
    async fn from_request_parts(parts: &mut Parts, _: &S) -> Result<Self, Self::Rejection> {
        let id = parts.extract::<Path<String>>().await?;
        AgentId::from_str(&id).map(ValidAgentId)
            .map_err(|_| ApiError::BadRequest { message: format!("Invalid agent_id: {id}") })
    }
}
```

**收益**: 消除所有 `unwrap_or("coder")` 默认值, 统一验证逻辑, 编译时保证安全。

---

### A-3: 存储层健康检查增强 (优先级: 中)

**现状**: `/api/health` 只检查存储层是否存在, 不检查实际可用性。

**目标**: 每个存储后端实现 `health_check()` 方法:
- L0/L1 Cache: 写入/读取一个 sentinel 值
- L2 LanceDB: 执行空查询
- L3 Graph: 执行 `?[] <- [[true]]` Datalog 查询
- L4 Files: 检查目录可写

---

### A-4: 请求限流与超时 (优先级: 中)

**现状**: 无请求限流, API 无超时保护。

**目标**: 添加 tower middleware:
```rust
// tower layers
.layer(TimeoutLayer::new(Duration::from_secs(30)))
.layer(RateLimitLayer::new(100, Duration::from_secs(1)))
```

---

### A-5: 配置热加载 (优先级: 低)

**现状**: `config::load_config()` 每次调用都重新解析 TOML 文件。

**目标**: 启动时加载一次, 存入 `AppState`, 提供 `/api/config/reload` 端点。

## 十、正面评价

1. **类型系统设计优秀**: Newtype IDs (AgentId, MemoryId, SessionId 等) 在编译期防止参数混淆
2. **Trait 抽象清晰**: 存储层完全通过 trait 隔离, 更换实现只需修改配置
3. **测试质量高**: 330 个测试覆盖行为而非实现, 与 CODING_STANDARDS.md 一致
4. **代码整洁度高**: 仅 2 处 TODO, 说明架构决策清晰, 无临时 hack
5. **Agent 隔离模型设计完善**: snapshot/restore 机制保证切换安全
6. **检索管道成熟**: BM25+ANN 混合召回 + 图谱扩散, 端到端可运行
7. **Dashboard 双模式 (Tauri/HTTP)** 设计灵活
